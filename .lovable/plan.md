## الهدف
إضافة صفحة **مبيعات كاش (POS)** للعملاء العشوائيين الذين يأتون إلى المحل بدون حساب مفتوح، مع تمييز فواتير هذه الصفحة في كل أنحاء النظام (الفواتير، التقارير، الحسابات، المخزون) وإمكانية فلترتها بشكل مستقل.

## الخطة

### 1. صفحة POS الجديدة `/pos`
- صفحة مبسطة مخصصة للبيع السريع، بدون اختيار عميل من قائمة.
- اسم العميل يُكتب يدويًا في حقل نصي (اختياري، الافتراضي: «عميل نقدي»).
- لا يتم إنشاء سجل في جدول العملاء — لا رصيد، لا كشف حساب، لا ديون.
- جدول بنود مبسط (منتج / كمية / سعر / إجمالي) بنفس تجربة جدول البنود الحالي (sticky header، zoom، اختصارات).
- الدفع نقدي افتراضيًا وكامل (الحالة `paid` مباشرة)؛ مع إمكانية تغيير طريقة الدفع (نقدي / تحويل) واختيار الحساب المستلِم.
- زرّان فقط: **«حفظ وطباعة»** و **«حفظ وجديد»**.
- لا يوجد قسم تغليف أو ترحيل في هذه الصفحة (يمكن إضافتهما لاحقًا من صفحة الفاتورة العادية إذا لزم).
- رابط في الشريط الجانبي باسم «مبيعات كاش» وأيقونة مميزة.

### 2. تمييز فواتير الـ POS في قاعدة البيانات
- إضافة عمود `source` (نص) إلى جدول `invoices` بقيم: `regular` (افتراضي) و `pos`.
- إضافة عمود `walk_in_customer_name` (نص اختياري) لتخزين الاسم اليدوي عندما لا يوجد `customer_id`.
- السماح بأن يكون `customer_id` فارغًا فقط عند `source = 'pos'`.
- الـ migration يضيف فقط الأعمدة + index على `source` للفلترة السريعة، بدون لمس البيانات الحالية.

### 3. ظهور فواتير الكاش في باقي الصفحات بشكل مميَّز
- **إدارة الفواتير (`InvoicesPage`)**:
  - عمود/شارة جديدة «نقطة بيع» بلون مميز (مثلاً `bg-primary/15 text-primary`) بجانب رقم الفاتورة.
  - فلتر علوي: «الكل / فواتير عادية / مبيعات كاش».
  - عند عدم وجود عميل: عرض `walk_in_customer_name` أو «عميل نقدي» بدل اسم العميل.
- **التقارير المالية** (تقرير المبيعات، تقرير الدخل، الفواتير اليومية):
  - تجميع فرعي خاص بـ«مبيعات الكاش» وعرضه كقسم/سطر منفصل في الملخص.
  - فلتر اختياري لاستبعاد أو عزل مبيعات الكاش.
- **الحسابات والمعاملات**:
  - عند الحفظ، إنشاء transaction بفئة `pos_sale` (بدل `invoice_payment`) ومرتبطة بالحساب النقدي المختار، مع وصف «بيع نقدي - <الاسم/رقم الفاتورة>».
  - في صفحة المعاملات: شارة/أيقونة مميزة لهذه الحركات.
- **المخزون**:
  - الخصم يتم بنفس آلية الفاتورة العادية (لا تغيير في المنطق).
  - في صفحة تتبع المخزون: ملاحظة الحركة `POS` مع لون مختلف لتمييز خصومات الكاش عن المبيعات الآجلة.

### 4. الطباعة
- استخدام نفس قالب الطباعة الحالي مع variant جديد `pos`:
  - رأس الفاتورة يكتب «إيصال بيع نقدي» بدل «فاتورة».
  - بدون قسم الحساب/الرصيد.
  - إخفاء حقول العميل التفصيلية، عرض الاسم فقط إن وُجد.
  - حجم A4 الافتراضي مع إمكانية لاحقًا لإضافة قالب إيصال ضيق إذا طلب المستخدم.

### 5. الفلاتر العامة
- في كل الشاشات التي تعرض فواتير (الفواتير، تقرير المبيعات اليومية، كشف حساب الفترة): إضافة خيار في الفلتر:
  - «الكل» / «فواتير حسابات (آجل/جزئي)» / «مبيعات كاش فقط».

### 6. الصلاحيات
- صفحة `/pos` متاحة لدور `admin` و`sales`.
- موظف الـ`sales` يستطيع إنشاء فواتير POS فقط إذا كان لديه `create_invoice` في `permissions`.

### 7. التحقق بعد التنفيذ
- إنشاء فاتورة كاش بدون اختيار عميل → تُحفظ بحالة `paid` وتظهر شارة «نقطة بيع».
- ظهورها في إدارة الفواتير والتقارير مع الفلاتر الجديدة.
- تأكيد أن المخزون يُخصم وأن الحساب النقدي يزيد بقيمة الفاتورة.
- تأكيد أن أرصدة العملاء لم تتأثر (لا يوجد customer_id).

## تفاصيل تقنية مختصرة
- **DB migration**: `ALTER TABLE invoices ADD COLUMN source text NOT NULL DEFAULT 'regular'; ADD COLUMN walk_in_customer_name text; CREATE INDEX invoices_source_idx ON invoices(source);` — مع GRANTs موجودة سلفًا، لا حاجة لسياسات جديدة.
- **ملفات جديدة**:
  - `src/pages/PosPage.tsx` (الصفحة).
  - `src/utils/posInvoice.ts` (helpers للحفظ + إنشاء transaction).
- **ملفات معدّلة**:
  - `src/App.tsx` (route + lazy).
  - `src/components/layout/AppSidebar.tsx` (رابط جديد).
  - `src/pages/InvoicesPage.tsx` (شارة + فلتر + اسم بديل).
  - `src/pages/Dashboard.tsx` (زر سريع اختياري «بيع نقدي»).
  - `src/utils/invoiceStatus.ts` بدون تغيير (نستخدم `paid`).
  - قوالب التقارير: `IncomeReportPage`, `DailyInvoicesReportPage`, `TransactionsPage`.
  - قالب الطباعة: `src/utils/printTemplate.ts` (variant `pos`).
